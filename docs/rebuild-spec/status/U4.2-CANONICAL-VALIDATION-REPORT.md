is consistent with the memory finding that the legacy Subhang shape is _closer_
to canonical but still structurally distinct.

## PART 3 — migrate-subhang-to-v2.mjs: coverage & refresh

**Transform output validates 100% against the CURRENT built domain:** dry-run =
**6,777 docs, 0 validation errors, 0 answer-field leaks** (tenant + code + 3
users + 3 memberships + 12 spaces + 191 storyPoints + 3,569 items + 2,997
answerKeys). Every Part-2 divergence is already handled:

| SUB001 divergence                                                                              | Handled by                                          |
| ---------------------------------------------------------------------------------------------- | --------------------------------------------------- |
| Timestamp→ISO (all docs)                                                                       | `isoDeep()` (D4 boundary, domain `toTimestamp`)     |
| `features.*Enabled`→canonical, `subscription.renewsAt`/`usage.resetAt` nullable                | `transformTenant` whitelist                         |
| `null`→`undefined` (phone/photoURL/consumerProfile), audit fields, `lastLogin` null            | `transformUser` / `transformMembership`             |
| `stats`/`ratingAggregate` rename, drop retake/price fields                                     | `transformSpace`                                    |
| `courseId` drop, `assessmentConfig` reshape, `createdBy/updatedBy/archivedAt`                  | `transformStoryPoint`                               |
| payload → discriminated union, embedded answers → deny-all `answerKeys/`, `meta.tags`→`labels` | `transformItem` / `buildQuestion` / `buildMaterial` |
| Shape A (`payload.questionData`) & Shape B (~60 payload-level-answer items)                    | `transformItem` merge `{...p, ...p.questionData}`   |

**Defensive hardening applied this task** (harmless: canonical passes through;
only future teacher-portal writes with legacy values are affected — current data
does not trigger them):

- `import { normalizeStoryPointType } from '@levelup/domain'` — storyPoint
  `type` now runs through the canonical `zLegacy*Read` normalizer
  (`'test'`→`'timed_test'`, per `legacy.ts` / AD-4). Previously passed through
  unmapped — a legacy `test` value would have failed strict.
- New `orderIdx(o)` helper: `orderIndex ?? order ?? 0`, applied to storyPoints,
  sections, and items — covers the legacy `order` field name if it ever appears.

Post-hardening dry-run re-verified: **6,777 writes, 0 errors, 0 leaks**
(unchanged).
