## 1. Authentication & Authorization

### Functional Requirements

| ID     | Requirement                                                                                                                                                                                                                                                                                                                                                                                                              | Priority |
| ------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | -------- |
| FR-001 | The system SHALL present a school code input form as the default login view (B2B Entry Point). On submission, the system SHALL validate the code against Firestore tenant records. If the tenant exists and has `status === 'active'`, the system SHALL display the school name and advance to the credentials step. If the code is invalid or the tenant is inactive, the system SHALL display an inline error message. | Must     |
| FR-002 | After successful school code lookup, the system SHALL present a tabbed login form with two methods: **Roll Number** and **Email** (B2B Credentials Login). The student enters their credential and password. On submit, the system SHALL call `loginWithSchoolCode(schoolCode, credential, password)`. On success, the system SHALL redirect to the `from` location (post-login redirect) or `/` by default.             | Must     |
| FR-003 | The system SHALL provide a consumer login form accessible via "Don't have a school code? Sign in as learner" link (B2C Email/Password Login). The form accepts email and password. On success, the system SHALL navigate to `/consumer`.                                                                                                                                                                                 | Must     |
| FR-004 | The consumer login form SHALL include a "Sign in with Google" button (B2C Google OAuth). On click, the system SHALL initiate Firebase Google OAuth flow. On success, navigate to `/consumer`.                                                                                                                                                                                                                            | Must     |
| FR-005 | The system SHALL provide a signup form with fields: Display Name, Email, Password (B2C Account Registration). On submit, the system SHALL call `createUserWithEmailAndPassword` followed by `updateUserProfile` to set the display name. On success, navigate to `/consumer`.                                                                                                                                            | Must     |
| FR-006 | The system SHALL implement a `RequireAuth` guard component (Route Protection) that: redirects unauthenticated users to `/login` with return-URL state; redirects consumer users (no `currentMembership`) attempting B2B routes to `/consumer`; renders "Access Denied" for authenticated users with wrong role; allows any authenticated user into B2C routes (no role restriction).                                     | Must     |
| FR-007 | On app mount, the system SHALL initialize Firebase Auth listener (Session Initialization). When `currentTenantId` is resolved, the system SHALL subscribe to tenant data in real-time via Firestore listener.                                                                                                                                                                                                            | Must     |
| FR-008 | The system SHALL provide sign-out functionality in both B2B (sidebar dropdown) and B2C (header button + dropdown) layouts. On sign out, the system SHALL clear auth state and navigate to `/login`.                                                                                                                                                                                                                      | Must     |

| FR-046 | The system SHALL support Text (Short Answer) with single-line text input and optional `maxLength` enforcement. Answer format: `string`. | FR-047 | The system SHALL support Paragraph (Long-Form) with resizable textarea (6 rows). Answer format: `string`. ## 8. Answer Evaluation & Grading

### Functional Requirements

| ID     | Requirement                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   | Priority |
| ------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------- |
| FR-057 | The following types SHALL be auto-evaluated server-side with deterministic logic (no AI): MCQ (exact option ID match, no partial credit), MCAQ (`(correct - wrong) / totalCorrect * maxScore`, partial credit), True/False (exact boolean match, no partial credit), Numerical (`abs(student - correct) <= tolerance`, no partial credit), Fill Blanks (per-blank string match case-optional, partial credit), Fill Blanks DD (per-blank option ID match, partial credit), Matching (per-pair ID match, partial credit), Jumbled (exact full-sequence match, no partial credit), Group Options (`(correct - wrong) / totalItems * maxScore`, partial credit). | Must     |
| FR-058 | The following types SHALL be evaluated by Gemini LLM (`gemini-2.5-flash` default) (AI-Evaluated Question Types): **Text** (LLM evaluation with rubric), **Paragraph** (LLM evaluation with `modelAnswer` and `evaluationGuidance`), **Code** (LLM code review, no execution, text-based), **Audio** (LLM speech/content evaluation), **Image Evaluation** (multimodal LLM image analysis), **Chat Agent** (LLM conversation quality assessment).                                                                                                                                                                                                              | Must     |
| FR-059 | After evaluation, the system SHALL display a `FeedbackPanel` showing: correctness icon + label (Correct/Partially Correct/Incorrect/Evaluated), score/maxScore points, progress bar with percentage, static explanation from question payload, overall comment from AI evaluation, strengths list (green), weaknesses list (red), missing concepts list (amber), and mistake classification (Conceptual / Silly Error / Knowledge Gap / None).                                                                                                                                                                                                                | Must     |
| FR-060 | The `QuestionAnswerer` SHALL render a collapsible "Previous Submission" section when `previousEvaluation` is available (Previous Submission Display).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         | Should   |
| FR-061 | In practice mode: submit button visible; input locks after submission (unless "Try Again" clicked); "Try Again" button resets the question for re-attempt; "Ask AI Tutor" button available (Practice Mode Controls).                                                                                                                                                                                                                                                                                                                                                                                                                                          | Must     |
| FR-062 | In test mode: submit button hidden (batch submission via test harness); input never locked by individual submission; `savedAnswer` prop used for session resume (Test Mode Controls).                                                                                                                                                                                                                                                                                                                                                                                                                                                                         | Must     |
